Core entity model

Node

Generic instance object. Example types may be world, region, organization, person, item, event, faction, or anything the host schema defines.

Edge

Typed connection between two nodes. Edges carry relation type, direction, metadata, and optional visual style.

Structure edge class

A special edge category for containment / location / composition. This powers container navigation and scene grouping, but still remains an edge, not a hardcoded tree parent field.

Visual projection

This split prevents visual edits from mutating business meaning accidentally. A person can move on the canvas without silently changing all memberships or relations.

Recommended schema contract

Host-defined type system
Node types define: display label, icon/shape, allowed fields, container capability, default style, validation rules
Edge types define: direction, whether multi-edge is allowed, whether it belongs to the structure class, validation rules, default line style
Templates define: starter node set, starter edge set, starter scene layout, optional field defaults

What this means for your writing scenario

1

No fixed world-org-person hierarchy

That writing structure becomes a preset schema + template, not a hardcoded product assumption.

2

Multi-membership remains valid

One person can belong to multiple organizations because membership is modeled through structure edges, not a single parent pointer.

3

Multiple scenes can exist later

The same graph can later power different views: world map, organization focus scene, social relation view, or chapter-specific subgraph.
