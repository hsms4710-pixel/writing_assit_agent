Interaction rules locked in

Drag always changes scene state

Node position, z-index, container-relative placement, and user-created clustering are persisted as scene data. Layout work is first-class user input, not throwaway decoration.

Drag into container creates a candidate command

The system infers a likely modeling action but does not silently mutate structural edges. A confirmation strip or context menu turns that candidate into an explicit command.

Inspector remains authoritative for exact semantics

Quick actions speed up modeling, but the inspector shows and edits the final set of memberships, roles, and relation properties.

Storage and sync boundary

Local-first, remote-ready
Local adapter responsibilities

load graph bundle
persist snapshots or op-log
recover unsynced edits
restore scene state fast
Remote adapter responsibilities

push commands or patches
fetch new versions
resolve revision mismatch
report sync status to host app

The core should expose storage ports and sync events, but it should not hardcode a backend. A writing app can store to local IndexedDB today and sync to its own service later.

Error handling and guardrails

1

Rejected candidate actions are non-destructive

If the user cancels the quick action after dragging into a container, only the scene placement stays. No hidden structural mutation should remain.

2

Invalid commands fail with explicit reason

If schema rules forbid a membership or relation, the quick action is disabled or returns a clear validation message.

3

Sync errors do not block local editing

Remote failure marks pending changes and exposes conflict state, but local work continues until the host app resolves sync.
